--------------------------------------------------------------------------------

1) matplotlib 백엔드를 Agg(비대화형)로 고정  ── modules/history.py 상단
   - 이유: 대화형 TkAgg 백엔드로 장시간 루프에서 figure를 반복 생성하면
     간헐적 segfault가 발생. 시뮬레이션은 PNG 저장 전용이라 Agg로 고정함.
   - 영향: `python main.py --plot True` 로 실행해도 마지막에 **대화형 그래프
     창이 더 이상 뜨지 않음**. 단, plot.png 는 그대로 저장되므로 그래프는
     파일로 확인 가능.
   - 창을 다시 띄우고 싶으면 history.py 의 `matplotlib.use("Agg")` 를 제거해야
     하지만 segfault 위험이 돌아옴(권장하지 않음).

2) 저장 그래프(plot.png)의 야간 음영 = "시계 기준 19:00~06:00"
   - 코드 안에 야간 정의가 두 가지로 서로 다름(아래 [B]-3 참고).
   - 그래프 음영은 이동 로직과 동일한 시계 기준을 사용. 전투 기준으로 바꾸려면
     history.py plot_team_strength_over_time 의 `_is_night` 함수 한 곳만 수정.
   - 음영 농도는 같은 함수의 `alpha=0.10` 값으로 조절.

[B] 코드 내 불일치 / 알려진 이슈 (이번에 손대지 않음)
--------------------------------------------------------------------------------

1) requirements 파일명 오타 (→ 해결됨)
   - 원래 파일명이 `requirments.txt`(오타)였음. requirements.txt 로 rename 하여
     커밋함(README 참조와 일치). 단, 내용의 numpy 핀은 아래 2) 참고.

2) numpy 버전 불일치
   - requirements 에는 `numpy==1.23.0` 으로 고정돼 있으나, 실제 wargame conda
     환경에는 numpy 1.26.4 가 설치돼 있음.
   - 주의: numpy 1.23.0 은 Python 3.13 을 공식 지원하지 않음. requirements
     그대로 새 환경을 만들면 설치 실패나 불안정(ABI) 가능성. 1.26.x 이상 권장.

3) "야간(night)" 정의가 두 군데에서 다름
   - 이동 로직 (troop.py calculate_movement_distance):
        hour = (13*60+55 + t)//60 % 24,  야간 = hour < 6 또는 hour >= 19
        → 실제 시계 19:00~06:00
   - 전투 로직 (troop.py assign_target / fire):
        is_night = 360 <= (t % 1440) <= 1080   → 대략 19:55~07:55
   - 두 정의가 약 1시간 어긋나고 길이도 다름. 통일할지 검토 필요.

4) build_cost_map 의 지형 비용 조회 (map.py)
   - `terrain_type = self.grid[i, j]` 로 가져온 값(이미 곱해진 float 비용)을
     다시 `terrain_cost.get(terrain_type, 1.0)` 로 조회함. grid 값(1.3, 3.5 등)이
     terrain_cost 의 정수 키(0~9)와 대체로 안 맞아 의도와 다르게 동작할 소지.
   - 1회성 계산(시작 시)이라 성능엔 무관하지만, 비용맵 값이 의도대로인지 확인 권장.

5) draw_troop_positions 의 `for troop in troop_list` (→ 수정 완료)
   - TroopList 는 iterable 이 아니라 `--save_frames True` 시 TypeError 로 즉시
     크래시했음. `troop_list.troops` 로 수정함.

[C] 보류해 둔 추가 최적화 (적용 시 시뮬레이션 결과가 미세하게 달라짐)
--------------------------------------------------------------------------------

현재까지 적용한 성능 최적화는 모두 "결과 불변"으로 검증됨(경로/웨이포인트 동일).
아래는 더 빠르게/가볍게 하려면 가능한 추가 옵션이나, 결과가 바뀌어 보류함:

   1) PATHFIND_COOLDOWN(현재 30분) 증가 → pathfinding 호출 횟수 감소(가장 큰 추가
      이득). 부대가 경로를 더 오래 따라가 전술 반응성은 약간 저하.
   2) Weighted A* (휴리스틱 가중치 강화) → 탐색 노드 수 급감, 경로는 근사 최단.
   3) 플로우 필드 float32 저장 → 캐시 메모리 절반(최대 262→131MB), 방향 정밀도
      미세 저하로 드물게 경로가 1픽셀 달라질 수 있음.
   4) 기록/프레임 빈도 축소 → status/visualization 기록 간격, save_tactics 빈도를
      낮춰 메모리·I/O 절감. 출력 CSV/프레임 시간 해상도가 거칠어짐.

   ※ flow_fields 캐시 상한(FLOW_FIELD_CACHE_MAX=32, troop.py)은 캐시 적중률이
     ~95%로 높으므로 메모리 때문에 낮추지 말 것(낮추면 재계산으로 느려짐).

[G] 학습/진행 중 관찰된 문제점 (issues log)

1) IPPO value loss가 큼(70~135) — 정책이 잘 학습된 BLUE에서도 높음.
   - 원인: 각 에이전트가 *로컬 관측*만으로 *팀 전체 승패(±1 공유)* 를 예측 → 분산 큰 credit-assignment.
   - 영향: RED 학습이 win ~20%에서 plateau, 불안정.
   - 대응(예정): MAPPO 중앙 critic(전역 state로 팀 가치 추정) → advantage 분산↓.

2) 매치업 비대칭: BLUE 버프(사거리×1.2/관측×1.2/피격×0.8)로 학습 시 BLUE ~95% vs RED ~20%.
   - 시나리오상 의도된 비대칭이지만, RED 학습 신호가 약함. 대응: 보상 재균형 또는 커리큘럼 검토.

3) 전투 난수가 전역 np.random(미시드)이라 에피소드 결과가 실행마다 다름.
   - 영향: 처리량/학습 곡선에 측정 노이즈, 재현 불가. [E]·베이스라인 비교(§10) 전 시드 통일 필요.

4) 체크포인트가 팀 무관하게 같은 파일로 덮어써짐 → (해결) 팀별 파일 분리 저장 ippo_<team>.pt.

5) 소규모 교전은 LOS로 접근 중엔 미관측, 근접하면 순식간에 종료 → 에피소드가 짧고 교전 프레임이 짧음.
